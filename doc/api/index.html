pip-services-runtime-node

Welcome to the Pip.Services Runtime for Node.js API docs page. These pages contain the reference materials for the version 0.0.

The documentation is organized by classes and interfaces. They are groupped into packages below:

Core

Contains interfaces for microservice components and other key abstractions.

  • IComponent - microservice component interface with component lifecycle definition
  • ComponentSet - set of microservice components with search capabilities
  • IComponentFactory - factory interface to find and instantiate components
  • State - component lifecycle states
  • AbstractComponent - abstract class for all microservice components
  • IDiscovery - interface for service discovery components
  • IConfigReader - interface for configuration readers
  • ILogger - interface for microservice loggers
  • LogLevel - enumeration for logging levels
  • ICounters - interface for performance counters
  • ITiming - interface to complete elapsed time measurement
  • ICache - interface for transient cache
  • IPersistence - interface for persistence (data access) components
  • IBusinessLogic - interface for business logic components (controllers and decorators)
  • IController - interface for business logic controller
  • IInterceptor - interface for interceptors that decorate controller and customize microservice behavior
  • IClient - interface for clients connectors to other microservices
  • IService - interface for API service (endpoint) that microservice exposes to its clients
  • IAddon - interface for microservice extentions

Config

Microservice configuration package

  • Category - category of components or configuration sections that are used to configure components
  • MicroserviceConfig - microservice configuration
  • ComponentConfig - component configuration section
  • ComponentDescriptor - component descriptor / identifier
  • Endpoint - service or client endpoint (address) configuration
  • Connection - database connection configuration

Build

Microservice construction and lifecycle management

  • ComponentFactory - factory to locate and instantiate components by component descriptors
  • FactoryRegistration - registration of components in component factory
  • DefaultFactory - factory to create default runtime components
  • Builder - microservice builder that creates components following configuration as a build recipe

Run

Microservice execution wrappers for various deployment platforms

  • LifeCycleManager - component lifecycle manager that inits, opens and closes a set of components
  • Microservice - microservice instance
  • ProcessRunner - runs microservices as a system process. It if also used for windows services, linux deamons or in docker deployments
  • SenecaPlugin - plugin for seneca microservice framework http://senecajs.org
  • LambdaFunction - wrapper for AWS lambda function http://aws.amazon.com/lambda

Discovery

Discovery components implementations. They used to register service addresses and dynamically resolve them for clients.

  • AbstractDiscovery - abstract class for all discovery components

Boot

Bootstrap configuration readers. They used to define types of microservice components and initialize their execution settings.

  • AbstractBootConfig - abstract class for all config reader components
  • FileBootConfig - reads microservice configuration from JSON file on disk

Logs

Logging components implementation. They used to report about microservice errors or log transactions performance by microservice.

  • AbstractLogger - abstract class for all logging components
  • NullLogger - NULL logger that doesn't do anything
  • ConsoleLogger - logger that outputs messages to standard output and error streams
  • CachedLogger - abstract logger that caches messages for batch output
  • LogEntry - individual log entry used by CachedLogger

Counters

Performance counting components implementations. They used to monitor microservice performance by measuring time intervals, counting calls, recording timestamps of key events or arbitrary values.

  • AbstractCounter - abstract class for all performance counters
  • NullCounter - NULL performance counter that doesn't do anything
  • LogCounter - performance counter that periodically dumps counters to log output
  • Counter - performance counter
  • CounterType - enumeration for performance counter types
  • Timing - callback class to complete elapsed time measurement

Cache

Caching components implementations. They used to store frequently accessed data in transient cache to avoid roundtrips to persistent storage for better performance.

  • AbstractCache - abstract class for all caching components
  • NullCache - NULL cache that doesn't do anything
  • MemoryCache - local in-memory cache to be used for testing
  • CacheEntry - cache value entry used by MemoryCache
  • MemcachedCache - connector to memcached distributed caching service https://memcached.org

Persistence

Persistence components implementations. They read/write data to a database or another storage.

  • AbstractDataAccess - abstract class for all data access components
  • MemoryDataAccess - in-memory data access to be used for testing
  • FileDataAccess - file data access that reads and writes data from/to JSON file on disk
  • MondoDbDataAccess - data access for MongoDB NoSQL database https://www.mongodb.com

Logic

Business logic controllers implementations. They encapsulate microservice business logic

  • AbstractBusinessLogic - abstract class for all controllers and decorators
  • AbstractController - abstract class for microservice controllers
  • AbstractDecorator - abstract class for all controller decorators

Clients

Client dependency components implementations. They provide user-friendly connectors to microservice API endpoints.

  • AbstractClient - abstract class for all client dependencies
  • DirectClient - client to make direct in-process calls to business logic within the same microservice container
  • RestClient - interoperable HTTP/REST client
  • LambdaClient - client to microservices deployed as AWS lambda functions http://aws.amazon.com/lambda
  • SenecaClient - client to microservice API endpoints via Seneca transports http://senecajs.org

Services

Service components implementations. They expose API endpoints to accept calls from microservice clients.

  • AbstractService - abstract class for all API services
  • RestService - interoperable HTTP/REST service
  • SenecaService - service endpoint accessed via Seneca transports http://senecajs.org
  • RequestDetector - detects client OS, browser and IP
  • RequestAuth - performs authorization of incoming requests
  • ResponseSender - sends proper HTTP responses with results and errors

Addons

Addon components implementations. They represent microservice extensions that take no part in business transactions, but provide additional services like reporting microservice health state, collecting usage metrics or performing random shutdowns for relience testing.

  • AbstractAddon - abstract class for all addons
  • SenecaAddon - addon that allows to control and share seneca instance across all components

Errors

Definition of error types which are thrown and processed by microservices.

  • ErroCategory - broad categories for microservice errors
  • MicroserviceError - base class for all microservice errors
  • UnknownError - internal or unknown error which shall not happen in well written code
  • BuildError - error occured during microservice build process
  • ConfigError - error in user-defined microservice configuration
  • StateError - error caused by calling operations out of order
  • ConnectionError - error happend during connect or disconnect to external service
  • CallError - unrecornized errors returned by external services
  • FileError - errors happend in local file operations
  • BadRequestError - error caused by a client submitted wrong operation parameters
  • NotFoundError - error caused by attempt to access object that doesn't exist
  • ConflictError - errors cause by version conflict in concurrent operations
  • UnauthorizedError - error due to absence of user credentials or insufficient access rights
  • UnsupportedError - unsupported or unimplemented errors

Data

Utility classes to support specific data processing patterns

  • IIdentifiable - interface for data objects identified by string ids
  • FilterParams - free-form query parameters
  • PageParams - query paging parameters
  • DataPage - envelop object to return paged query result
  • IdGenerator - generator of unique object ids
  • TagsProcessor - extracts from data objects and processes search tags

Validation

Data validation schemas

  • Schema - validation schema for data objects
  • IValidationRule - interface for schema validation rules
  • PropertySchema - validation schema for specific object properties
  • IPropertyValidationRule - interface for property validation rules

Commands

Implementation of command pattern for business operations. It allowes to easily intercept, delegate and override operations in chain of component calls that interact to fulfil business requests

  • ICommand - command interface to implement command pattern for business operations
  • Command - command to wrap a business method
  • ICommandInterceptor - interceptor to intercept calls to multiple commands
  • InterceptedCommand - command to inject inceptors in command call chain
  • CommandSet - command set that handles command requests
  • TimingIntercepter - intercepter to track commands execution time
  • TracingIntercepter - intercepter to log trace messages for executed commands

Portability

Package containing cross-language and cross-platform abstractions and utility functions.

  • DynamicMap - language-independent hashmap to store hierarchical dynamic data
  • Converter - value type converter to convert objects to strings, numbers, dates and maps
